## 0. 4대 설계 원칙 (왜 이렇게 하나)
1. **척도 방향 통일 (ESAS)** — 모든 증상 0~10, **0=없음/최상, 10=최악/심함**. 방향 혼동 제거. 개선=점수↓, Δ=사전−사후(양수면 호전). 국제 검증 척도(암 증상 자가평가) → 임상 비교·정책 근거 가능.
2. **처방-측정 연결** — 모든 측정은 `prescription_id`에 묶임 → 그날 숲·피톤치드지수·날씨와 연결 → "피톤치드↑ 날 Δ효과↑?" 검증. ★ 프로젝트의 핵심 루프.
3. **페어링 키 고정** — participant_id(앱=로그인/전화). 사전·사후·추적을 한 키로 묶어야 Δ 계산. 이름 매칭 금지.
4. **정적 1회 + 린 코어** — 프로필(암 정보 등)은 가입 시 1회. 매 방문은 핵심만 2분. 연구 상세는 선택 모듈.

---

## 1. 가입 시 1회 — 정적 프로필
`participants` 테이블. 매 방문 재질문 안 함.
- 성별 [남 / 여]
- 연령 (만)
- 암 진단 연도
- 암 종류
- 치료 상태 [완료 / 치료 중 / 기타]
- (선택) 평소 수면시간, 주간 운동빈도 [없음/1~2회/3회+], 복용 보조제

## 2. 사전 설문 (방문 전, ~2분)
### 2-A. 추천 구동 입력 (→ 엔진)
| 문항 | 옵션 | 엔진 매핑 |
|---|---|---|
| 이번 방문에서 가장 개선하고 싶은 것 (1개) | 수면 / 피로 / 스트레스 / 통증 / 전반 컨디션 | `goal` |
| 무엇이 더 중요하세요 (1개) | 가까운 곳이 좋다 / 특별한 경험이 좋다 | `user_type` (comfort/explorer) |
| 출발 위치 | 현재위치 또는 주소 | `lat`,`lon` (거리 점수) |

### 2-B. 기준선 — ESAS 0~10 (★ 사후와 동일 항목, 페어링 핵심)
**안내문구: "지금 이 순간, 각 항목이 얼마나 심한지 0~10으로. 0=전혀 없음/매우 좋음, 10=매우 심함/매우 나쁨"**
- 피로 (0=없음 ~ 10=매우 심함)
- 통증
- 수면 곤란(불면)
- 우울감
- 불안감
- 스트레스
- 전반적으로 힘든 정도 (0=매우 좋음 ~ 10=매우 힘듦)

> 방향 통일이 핵심. 기존의 "수면의 질/전반 건강(높을수록 좋음)" + "피로도/스트레스(모호)" 혼용을 전부 **심함=높음**으로 정렬.
> 한국어 검증판 ESAS-r 문구를 그대로 채택하면 연구 신뢰성 최대.

## 3. 사후 설문 (방문 직후, ~2분)
- (자동) participant_id + **prescription_id** (어느 방문인지)
- **기준선 재측정**: 2-B의 ESAS 항목 **그대로 동일** → Δ 계산
- 체감 변화 (1개): 매우 좋아짐 / 조금 좋아짐 / 변화 없음 / 나빠짐
- 변화의 가장 큰 이유 (1개): 자연환경 / 프로그램 활동 / 휴식 / 사람·분위기
- (간단) 체류시간(분), 동행 [혼자 / 가족 / 지인] — 교란변수 보정용
- (선택) 프로그램 만족도 0~5
- 향후 참여 의향 [매우 있음/있음/없음] + 맞춤 리포트 신청(연락처)

---

## 4. 3일 후 추적 (푸시 알림, ~30초)
효과 **지속성** 측정. 핵심 2개만.
- 수면 곤란(불면) 0~10
- 피로 0~10
> "그날만 좋았나 vs 3일 갔나" = 진짜 효과의 증거.

## 5. DB 스키마 (효과 검증용)
```sql
participants(
  id pk, sex, age int, cancer_dx_year int, cancer_type text,
  tx_status text, sleep_hours numeric, exercise_freq text,
  supplements text, created_at timestamptz)

prescriptions(
  id pk, participant_id fk, created_at timestamptz,
  goal text, user_type text, lat float8, lon float8,
  pick_seq int,                      -- 추천된 숲(healing_forests.seq)
  phyto_index_at_visit numeric,      -- 그날 그 숲의 피톤치드 지수  ← 검증 핵심
  pm25_at_visit numeric, temp numeric, humidity numeric, wind numeric,
  payload jsonb)

measurements(
  id pk, prescription_id fk, participant_id fk,
  phase text,                        -- 'pre' | 'post' | 'd3'
  fatigue int, pain int, insomnia int, depression int,
  anxiety int, stress int, wellbeing int,   -- ESAS 0~10 (동일 항목)
  change_overall text, change_reason text,  -- post 전용
  stay_min int, companion text,             -- post 전용(교란변수)
  at timestamptz)
```

### 핵심 분석 쿼리 (검증 루프)
- **방문 효과**: `Δ = pre - post` (항목별). 양수 = 호전.
- **지속성**: d3 vs pre.
- **★ 인과 신호**: `phyto_index_at_visit` ↔ `Δstress`(또는 Δfatigue) 상관 → "피톤치드 높을수록 효과 큰가". 이게 추천(처방)이 효과로 검증되는 증거.
- **종단**: 동일 participant 반복 방문 추세.

## 6. 대회 데모용 (D-6)
- 실데이터는 없으니 **더미 페어 데이터**로:
  - 5~10명 × 사전/사후 ESAS → "평균 스트레스 6.8 → 4.2, 수면곤란 7.1 → 4.5" 식 막대 비교
  - phyto_index ↔ Δstress 산점도 한 장 → "근거 엔진" 스토리 완성
- 발표 멘트: "국제 검증 척도(ESAS)로 측정, 처방의 환경 조건과 연결해 효과를 데이터로 검증. 현재는 시뮬레이션, 실사용자 데이터 축적이 다음 단계."

```jsonc
{ "prescription_id": "...", "phase": "pre",   // pre|post|d3
  "fatigue": 7, "pain": 3, "insomnia": 8, "depression": 4,
  "anxiety": 5, "stress": 7, "wellbeing": 6,
  "change_overall": null, "change_reason": null,
  "stay_min": null, "companion": null }
// → 저장 후 phase=post일 때 pre와 묶어 Δ 반환(리포트용)
```
